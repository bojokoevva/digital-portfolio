Dron Teq – Canada Expansion Change Request

Change request proposal for adapting the Dron Teq drone services platform from U.S.-only operations to a fully supported Canadian market, including new requirements, use cases, and architecture.

Context & goal

Dron Teq provides drone flight services and data analysis and originally operated only within the United States. As the business expands into Canada, the existing system must be adapted for new regulations, payment methods, geography, and bilingual communication.

Our goal in this project was to craft a formal change request that extends the current architecture for Canadian operations: defining updated operational and security requirements, new and modified use cases, and a design and implementation strategy that keeps the system coherent instead of building a separate product.

Updated requirements

We began by extending the existing Dron Teq requirements catalog to cover Canadian operations rather than rewriting it from scratch.

  • Expanded operational & performance requirements, including web/mobile access, cloud storage via Amazon S3, and support for at least 100 simultaneous users and 50 concurrent video streams with response times under 7 seconds. :contentReference[oaicite:1]{index=1}
  • Strengthened security requirements with encrypted payment flows, secure upload/download of drone videos and analyses, and explicit access control for high-value client data.
  • Added cultural & political requirements for Canada: French language support, Canadian address and postal formats, currency conversion to CAD, GST/HST/PST handling, and local payment methods such as Interac. :contentReference[oaicite:2]{index=2}
  • Introduced legal constraints like PIPEDA-compliant data storage and region-aware handling of flight data in S3 buckets.

Use cases & process logic

My main contribution in the analysis phase was updating core use cases so that they work consistently across both U.S. and Canadian operations.

  • Notify pilots of new flight request (U.S. & Canada) – extended the logic to detect country from coordinates, build a list of nearby pilots, apply Canadian tax rules, and send bilingual notifications that show province/territory and bidding window countdowns. :contentReference[oaicite:3]{index=3}
  • Pilot submits bid (U.S. & Canada) – added currency conversion to CAD, localization of language based on pilot preference, and flags for tax adjustments on Canadian bids.
  • Select winning flight bid – updated the manager’s workflow to handle CAD pricing, tax-inclusive bids, drone capability checks, and bilingual notifications to selected and non-selected pilots as well as clients. :contentReference[oaicite:4]{index=4}

For each use case we captured preconditions, normal and exception flows, and a summary inputs/outputs table so developers can see how data moves between datastores and user interfaces.

Architecture, design & implementation plan

In the design and implementation phases I helped translate the requirements into concrete changes to the architecture, UI, and operational processes.

  • Proposed architecture adjustments such as integrating Environment Canada weather APIs and Canadian GIS data for airspace maps, plus a regional selector that toggles between U.S. and Canadian behavior. :contentReference[oaicite:5]{index=5}
  • Specified UI updates for bilingual language switching, CAD pricing, dynamic tax display, and Canadian address formats.
  • Designed program changes for Transport Canada pilot certification checks, Canadian bidding logic, Interac / Canadian bank payments, and PIPEDA-compliant data storage separation.
  • Helped outline a testing strategy covering bilingual UX tests, compliance/security audits, performance testing with Canadian modules enabled, and regional simulations across provinces/territories. :contentReference[oaicite:6]{index=6}

Impact & what I learned

This project showed me what it looks like to scale a real system from one country to another while staying honest about regulations, language, and payment realities.

  • Practiced thinking in terms of change requests: modifying an existing system responsibly instead of designing in a vacuum.
  • Strengthened my ability to connect geospatial data, regulations, language, and payments into one coherent set of requirements and use cases.
  • Gained experience leading a team through multiple phases (planning → analysis → design → implementation & testing) and keeping the documentation consistent from section to section.

Dron Teq reinforced how much I enjoy cross-border systems problems: where technology, regulation, and user experience all have to align in a single design.
