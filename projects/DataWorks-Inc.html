DataWorks Inc. – Employee & Project Analytics

Designing a compact PostgreSQL database for a fictional data consulting firm that tracks employees, departments, projects, and the hours spent on each engagement.

PostgreSQL SQL queries M:N relationships Analytics
ISM 4212 – Database Design & Administration University of South Florida · Prof. Nasir Uddin
Role Data modeler, query designer, and implementer
Context & goal Business scenario

A mini consulting firm where each employee can work on many projects, and each project can involve many employees — exactly the kind of M:N structure that appears in real organizations.

DataWorks Inc. is a fictional data consulting firm that needs to track its staff, departments, client projects, and the hours each consultant spends on those projects. The narrative describes how employees rotate between projects, which projects are led by whom, and how budgets and work hours should be monitored.

My goal in this project was to turn that narrative into a clean relational schema that:

  • captures all business rules from the scenario,
  • supports analytics on budgets and hours, and
  • feels realistic enough to extend for future cases.
Domain Consulting · internal operations · project staffing
Focus M:N relationships · join tables · practical SQL
Tools & tech PostgreSQL · DBeaver · ERDPlus · SQL scripts
My role Designed schema, loaded sample data, wrote analytical queries
Technical design Relational schema

Core tables for employees, departments, projects, and the bridge table that captures who works on what, for how long, and in what role.

I started by identifying the main entities in the narrative: Employee, Department, Project, and the M:N relationship Employee_Project that stores the hours and whether someone is a project leader.

  • Defined Department with attributes such as department name and location.
  • Modeled Employee with names, department IDs, and other key attributes.
  • Created Project with descriptions and budgets, capturing the consulting work DataWorks delivers.
  • Implemented Employee_Project as the join table, including hours, start dates, and a leader flag.

The schema is intentionally compact but expressive: with only a few tables, it already supports many analytical questions (who works on which project, where departments are located, which projects use the most hours, etc.).

I used PostgreSQL and DBeaver to create the tables, define keys and constraints, load sample data, and then explore the dataset through increasingly complex queries.

Queries & analytics SQL practice

From simple joins to grouped summaries showing project budgets and total hours per engagement.

Example questions I answered with SQL:

  • Which employees work on which projects?
    Joining employee, department, and employee_project to list names, project descriptions, hours, and leaders.
  • What are the budgets for each project?
    Selecting from project and ordering by budget to see where the largest engagements are.
  • How many total hours were spent per project?
    Grouping employee_project by project and summing hours to compare workload across initiatives.

These queries simulate the reporting a manager at DataWorks Inc. would want: understanding where people are assigned, which projects are consuming the most effort, and how that aligns with budgets.

Because the schema is well-normalized, the SQL stays surprisingly readable—even when combining multiple joins and aggregations.

Impact & what I learned Takeaways

A small project that became a playground for practicing joins, constraints, and analytics that feel like real consulting work.

DataWorks Inc. was a great “mini-world” for practicing how real organizations use relational databases. I strengthened my intuition about M:N relationships, learned how to design join tables that carry extra business meaning (hours, leader flags), and practiced writing queries that managers actually care about.

This project also prepared me for larger cases like hospital databases and networked systems, because the core ideas—clean schemas, clear constraints, and well-documented SQL—scale up directly.
